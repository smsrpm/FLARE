Preface
FLARE is a compact reactor safety analysis environment for rapid accident-response calculations, sensitivity studies, uncertainty runs, risk-informed screening, and technical demonstrations. This User’s Manual explains how to install, start, configure, run, and interpret the tools that make up FLARE. It is written for an analyst who understands the broad purpose of design-basis and beyond-design-basis calculations but needs practical guidance on how the FLARE user interface, input files, output products, and supporting utilities fit together.
The companion Code Theory Manual describes the mathematical models. This manual is deliberately operational. It focuses on what the user must do, what the software will produce, what assumptions must be made explicit, and how to recognize common mistakes. Where a model limitation affects the way a case should be set up or interpreted, that limitation is repeated here in user-facing language rather than left only in the theory documentation.
Note: FLARE is a reduced-order screening and methods-demonstration tool. It should not be used as a substitute for a plant-specific licensing evaluation model, a formally assessed containment code, or a detailed radiological consequence analysis code. All results should be interpreted in the context of the simplifications described in Chapter 11.
Chapter 1 — Installation and Startup
This chapter describes the minimum software environment needed to run FLARE, the expected file organization, and the normal startup sequence. A reliable installation is important because FLARE tools share input workbooks, output folders, configuration files, and worker-process status files. Most startup problems are caused by missing Python packages, running from the wrong working directory, or file locking by synchronization services such as OneDrive.
1.1 Requirements
FLARE requires Python 3.11 or 3.12. The recommended installation is a Python virtual environment dedicated to FLARE so that package versions remain isolated from other projects. The complete set of required packages is:
pip install streamlit plotly openpyxl pandas numpy XSteamPython scipy reportlab anthropic
Package | Purpose

plotly | Interactive time-history plots in the PWR Simulator and Plant Analyzer.
openpyxl | Reading and writing Excel input and output workbooks.
pandas | Tabular data processing for CSV output, UA results, and risk tables.
numpy | Numerical arrays and mathematical operations throughout the simulation engine.
XSteamPython | Water and steam thermodynamic properties (pressure-enthalpy surface).
scipy | Root-finding (brentq) used in the critical-flow and steam-property solvers.

anthropic | API client for AI Event Narratives and the FLARE Assistant (optional but recommended).

1.3 Starting FLARE
FLARE is launched as a Streamlit application. Start from the FLARE folder. On a system where Streamlit is available on the path:
streamlit run flare_home.py
On Windows with a virtual environment, activate the environment first:
cd C:\path\to\FLARE
flare_env\Scripts\activate
streamlit run flare_home.py
A convenience launcher script for Windows is provided at Install\start_streamlit.ps1. This script checks for the virtual environment, installs missing packages if needed, and starts Streamlit automatically.
The Home page opens in a browser, typically at http://localhost:8501. From the Home page, launch the PWR Simulator, Model Editor, Plant Analyzer, Uncertainty Analysis, Risk Assessment, or FLARECON workflows using the tool tiles.
Note: If the FLARE folder is inside OneDrive or another cloud-sync folder, pause sync before running simulations. OneDrive can lock Excel input files or output folders and cause PermissionError messages during input reading or output writing.
1.4 Platform Support
FLARE detects the runtime platform at startup and adjusts its behavior accordingly. The simulation engine fully supports Windows, macOS, and Linux desktops. It can also run on iOS (Pyto, Pythonista) and Android (Termux) in a restricted mode that skips GUI display and Streamlit; in that mode, flare_sim.py can be called directly from the command line and outputs a CSV file.
On headless Linux servers without a display, FLARE automatically uses the Agg matplotlib backend so figure-generation code does not fail. The preflight check writes a sentinel file (Runtime/.preflight_ok) after the first successful package verification; subsequent launches skip the slow package check and start faster.
Chapter 2 — FLARE Home
The FLARE Home page is the navigation and configuration hub. It provides a single entry point for launching individual tools and for configuring the FLARE Assistant. A typical workflow begins on the Home page, proceeds to the Model Editor or PWR Simulator, and then returns to the Home page to launch post-processing or supporting analysis tools.
2.1 Navigation Tiles
Each tile on the Home page represents a FLARE tool. Clicking a tile opens that tool in the same browser session. Each tool includes a Home button in its sidebar so the user can return without restarting Streamlit. If a tool file is missing from the FLARE folder, the Home page displays a user-friendly message instead of a Python traceback.
2.2 FLARE Assistant
The FLARE Assistant is an AI chat interface for answering questions about FLARE models, parameters, output columns, and common workflows. It draws on the Code Theory Manual, User’s Manual, and supporting technical reference material. The Assistant is most useful for questions such as “what does this parameter do,” “where is this output produced,” or “how should I set up this class of case.”
The Assistant requires an Anthropic API key. The preferred approach is to enter the key on the FLARE Home sidebar. FLARE writes the key to Runtime/flare_config.txt in the working folder. The same file may also be created or edited manually:
ANTHROPIC_API_KEY = sk-ant-...
ANTHROPIC_MODEL = claude-sonnet-4-6   # optional; omit to use the default model
Note: API use requires an internet connection and may incur charges under the Anthropic account associated with the key. Do not commit a personal API key to a shared code repository.
Chapter 3 — Model Editor
The Model Editor is the recommended way to inspect and modify FLARE input workbooks without directly editing every cell in Excel. It reads CaseName_in.xlsx files, groups parameters into labeled sections, displays tooltips from inline comments, and writes changes back to the original workbook.
3.1 Overview
The Model Editor treats the input file as a structured model, not merely a spreadsheet. RCS parameters and FLARECON parameters are displayed separately when both are present. Each parameter group appears as a collapsible section with labeled fields on the left and editable inputs on the right. This view is particularly useful when a workbook contains many parameters or when verifying comments, defaults, and section labels.
3.2 Creating a New Input File
To create a new case, select New Case in the sidebar and enter a case name following the CaseName convention (no spaces, no special characters). The editor creates CaseName_in.xlsx and populates it with a default RCS command block and a default time-series table. Run the new workbook once as a single interactive case before using it in a batch, UA, or Risk run to confirm the input is valid.
3.3 Editing Existing Parameters
When an existing case is selected, the editor loads the workbook and organizes each labeled parameter group as a collapsible section. Tooltips are built from inline comments in the RCS sheet or from the description column in the FLARECON sheet. After editing, click Save RCS Model or Save FLARECON Model to write changes back to the workbook.
Note: Saving overwrites the original workbook. Create a backup copy before editing when preserving a baseline case is important.
3.4 FLARECON Tab
If the workbook contains a sheet named exactly FLARECON, the Model Editor displays a separate FLARECON Model tab. The FLARECON tab uses plain-text section headers such as DESIGN PARAMETERS and INITIAL CONDITIONS instead of the #-prefixed headers used in the RCS sheet. Keeping containment inputs in the same workbook as the RCS model ensures that the PWR Simulator can run the integrated calculation automatically.
3.5 Import, Resample, and Full Table Editing
The Import/Resample function reads an existing case and interpolates the time-series table to a new time grid. This is useful when a long transient needs a coarser control table or when an RIA calculation requires a very fine early-time reactivity table. The Full Table Editor exposes the entire time-series table as an editable dataframe. The time column must start at zero and increase monotonically.
Chapter 4 — PWR Simulator
The PWR Simulator is the primary execution and results-review interface. It runs individual cases, applies temporary sidebar overrides, launches FLARECON automatically when a containment sheet is present, displays calculated time histories and scalar metrics, and generates AI-supported event narratives and batch reports.
4.1 Overview and Typical Workflow
A normal interactive analysis begins by selecting a case, reviewing the input assumptions, optionally applying sidebar overrides, and running the simulation. When the run completes, review the scalar tiles for obvious failures, then inspect the time-series plots in sequence with the event chronology. For integrated RCS+containment cases, review both the RCS Results tab and the Containment Response tab before drawing conclusions.

4.2 Running a Case
Select a case from the Case dropdown in the sidebar.
Review the input assumptions. Apply sidebar overrides only for temporary sensitivity calculations.
Select Fast mode if figures are not needed. Fast mode is appropriate for debugging, UA, Risk, and batch runs.
Click Run Simulation. The progress bar shows RCS simulation time advancing. If a FLARECON sheet is present, containment status messages appear after the RCS simulation completes.
Review the Results tab. If CaseName-CON.csv was produced, review the Containment Response tab.
Note: At run start, FLARE makes a private temporary copy of the input workbook so the user may leave the original open in Excel without causing a PermissionError. The original file is not modified. The exact copy used for the run is archived in the timestamped run folder.
4.3 Execution Modes, Run Folders, and Worker Processes
Interactive single-case execution runs within the Streamlit session and is best for development, debugging, and interpretation. Long-running operations such as Run All Cases use durable workers. The UI writes a configuration file, starts a child process, and then monitors status files in a timestamped run folder named sim_CaseName_YYYYMMDD_HHMMSS. The worker may continue even if the browser disconnects.
When diagnosing a failed run, inspect the run folder first. Per-case console logs typically identify missing inputs, property-lookup failures, or output-write problems. The run folder preserves the archived input snapshot, the output CSV, any Excel output, individual case logs, and the final status JSON.
4.4 Results Tab
The Results tab combines scalar metric tiles with interactive time-history plots. The scalar tiles are useful for quick screening, but the plots are essential for physical interpretation. For example, a high peak cladding temperature should be reviewed together with vessel level, break flow, ECCS injection, DNBR, heat-transfer coefficient, and core power to determine whether the result is driven by blowdown, delayed injection, dryout, or decay heat. The unit-system selector changes display units but does not alter the underlying CSV data.
4.5 Containment Response Tab
When a CaseName-CON.csv file is present in the run folder, a Containment Response tab appears. This tab displays peak containment pressure, gas temperature, hydrogen concentration, steam concentration, sump level, and flammability status. Time-history plots show the competing effects of break source, wall heat removal, floor and sump heat capacity, spray heat removal, condensation, and hydrogen addition. The Shapiro diagram, when generated, shows the path of the steam-air-hydrogen mixture through the flammability envelope.
Note: The containment calculation is one-way coupled. RCS output drives FLARECON, but containment back-pressure does not feed back into the RCS break-flow calculation.
4.6 AI Event Narrative and Report Outputs
The AI Event Narrative converts the numerical results into a structured, section-by-section technical narrative. The narrative is data-grounded: FLARE passes selected scalar metrics, sequence-of-events data, initial and final conditions, source-term tables, dose tables, containment results, and unit-system preference to the language model. Each section of the narrative is generated by a separate API call and displayed as it completes, so the user can see progress in real time. After all sections are assembled, a progress bar and status indicator mark completion and the page refreshes to show the final stored report.
The number and scope of sections is controlled by the Narrative detail slider. At detail 0.00, the narrative is a short abstract covering only the initiating event and plant response. At detail 0.15, a safety system and thermal-hydraulic response section is added. At 0.25, a fuel integrity and core thermal response section is added. At 0.40, a containment response section is added when FLARECON data are present. At 0.55, a source term and radiological consequences section is added when source-term data are present. At 0.75, a review observations and model caveats section is included. At detail 1.00, all available sections are generated at their maximum allocated word target. For benign or verification cases with no fuel damage, FLARE condenses the narrative to a short three-section structure regardless of the detail slider, and sections covering fuel integrity or source term are suppressed when their data are absent or zero.
The Build tables only option creates the Sequence of Events and Initial/Final Conditions tables without an API call. This is useful when deterministic tabular output is needed or when an API key is not available. Download as Word document creates an individual report combining the narrative, tables, and case metadata. In Run All Cases mode, individual reports and plots are compiled into a final PDF report.

Chapter 5 — Input File Format
FLARE input files are Excel workbooks that combine declarative parameter assignments with optional time-dependent forcing tables. Correct input organization is critical because the same workbook may be used by the Model Editor, PWR Simulator, Uncertainty Analysis, Risk Assessment, Plant Analyzer, and FLARECON. This chapter explains the workbook structure and gives practical guidance for maintaining cases that remain readable and reproducible.
5.1 RCS Command Block
The RCS command block occupies the top rows of the first worksheet. Each row is a Python-style assignment statement. Inline comments begin with # and should be used to preserve engineering units, modeling intent, and important assumptions. The command block ends when the first numeric value appears in column A, which marks the start of the time-series table.
FLARE parses the command block by executing each assignment using Python’s exec() in an isolated namespace. The parser handles common formatting issues automatically:
Unicode minus signs and en-dashes are converted to ASCII hyphens.
Trailing unit annotations such as kPa or m³ are stripped before numeric evaluation.
Thousands-separator commas in plain numeric values (e.g., -3,000) are removed.
An example RCS command block for a large-break LOCA case:
# CaseLBLOCA - Large Break LOCA from full power
diameter_break = 0.22          # m, cold-leg break
pressure_kPa   = 14820         # kPa, initial pressure
total_power    = 575           # MWt
core_flag      = 1
5.2 Time-Series Table
The time-series table provides user-specified transient forcing functions. It starts after the command block, beginning with a row whose first cell starts with the word “Time” (case-insensitive). The parser confirms the header by requiring that at least one numeric value follows within the next three rows.
Column | Header | Purpose
A | Time (s) | Simulation time. Must start at zero and increase monotonically.
B | Structure Q [MW] | Generic heat-structure input when cond_flag is enabled.
C | Decay Heat [MW] | Decay heat override. Zero means use the built-in ANS-5.1 decay heat model.
D | SG Q [MW] | Steam-generator heat removal override when sg_table_flag = 1. Positive = normal SG removal; negative = SG adds heat (MSLB overcooling).
E | CVCS Flow [kg/s] | Net CVCS makeup or letdown adder on top of the baseline cvcs_kgs steady-state flow.
F | rho_ext [pcm] | External reactivity insertion for RIA or ATWS studies.
Table 5-1. Time-series table column definitions.
Up to three consecutive blank or non-numeric rows are tolerated in the table before parsing stops. This allows the user to insert visual separator rows without corrupting the input.
5.3 FLARECON Sheet
To add containment response analysis, add a second worksheet named exactly FLARECON to the input workbook. FLARECON uses key = value entries like the RCS sheet but uses plain-text section headers instead of #-prefixed headers. The PWR Simulator detects the sheet automatically and runs FLARECON after the RCS simulation has completed.
DESIGN PARAMETERS
air_volume_m3 = 10000          # m³ free gas volume
floor_area_m2 = 435.6          # m² floor area
INITIAL CONDITIONS
P0_kPa = 101.325               # kPa initial pressure
T0_C   = 27.0                  # °C initial temperature
5.4 Parameter Classes
Treating a FLARE input workbook as a declarative model rather than a collection of tuning knobs produces cases that are easier to review and prevents accidental compensation of one modeling error by another.
Parameter class | Examples | How the user should treat it
Geometry | vessel area, loop volume, heated length, pin diameter | Change only when representing a different plant or scaled model.
Initial conditions | RCS pressure, core exit temperature, accumulator pressure | Check first when a transient starts from the wrong state.
Boundary conditions | break diameter, SI flow, SG secondary temperature | Use to define the initiating event and mitigating systems.
Model flags | core_flag, pja_flag, sg_table_flag | Document flag choices; they alter model behavior fundamentally.
Numerical controls | timestep, endtime, output frequency | Use to control integration and reporting, not to tune physics.

5.5 Input Integrity Practices
Before using a case in UA, Risk, or Run All Cases, run it once interactively. Confirm that the initiating event occurs at the intended time, safety systems actuate as expected, and the final state is physically plausible. Keep case names concise and descriptive. Avoid spaces and special characters in file names when the case will be used in batch runs or shared across systems.
Note: When using sidebar overrides, remember that the workbook is not changed. Record important override values in the case notes, report narrative, or a separate configuration record.

Table 5-4. NOTBADTRAD dose model parameters.
The burnup, dispersion-override, and calibration parameters above support benchmarking a case against a site-specific licensing calculation. nbt_burnup_gwdmtu scales the default NUREG-1465-based group inventories to a plant-specific maximum average (or, for large PWRs with low-leakage core loading, peak assembly) burnup; for example, nbt_burnup_gwdmtu = 51 corresponds to the 51 GWd/MTU surrogate-plant burnup in the Clinch River ESPA SSAR (Rev. 2, Section 15.2).
nbt_chi_q_eab, nbt_chi_q_lpz, and nbt_chi_q_cr allow a site-specific atmospheric dispersion factor (e.g. from PAVAN or ARCON96) to be supplied directly for the corresponding receptor, in place of the value computed from nbt_wind_speed_m_s, nbt_stability_class, and the nbt_distance_*_m parameters. This is the recommended approach when comparing NOTBADTRAD results to a plant FSAR or SSAR dose calculation, since the embedded Pasquill-Gifford model is an approximation that becomes less reliable at longer receptor distances and for averaging periods longer than a few hours.
nbt_calibrate is a final multiplier on the EAB, LPZ, and control-room TEDE results and on the corresponding group-level and TEDE-versus-distance outputs. It does not affect inventories, release fractions, or any intermediate quantity. Because it can mask an underlying modeling difference if used carelessly, any value other than 1.0 should be documented in the case file together with the comparison case and result that motivated it.

The Plant Analyzer displays an SVG representation of the primary system and, when available, an animated containment dome. Chapter 7 — Uncertainty Analysis
The Uncertainty Analysis tool performs Monte Carlo propagation of selected input uncertainties. Its purpose is to quantify how uncertain parameters affect scalar figures-of-merit and time-history envelopes. The tool also supports Wilks-style one-sided tolerance interpretations when the sampling plan is defined before the run and the samples are independent realizations of a fixed uncertainty model.
7.1 Overview
A UA run starts from a base case and generates N sampled cases by perturbing selected inputs according to distributions specified in Runtime/flare_ua_variables.json. Each sampled case is run by the FLARE simulation engine. The tool then collects scalar metrics, time histories, sample values, statistics, envelope plots, CDF plots, and correlation measures.
7.2 Setting Up a UA Run
Select a base case that has already run successfully as a single deterministic case.
Choose the number of samples. For a one-sided first-order 95/95 Wilks estimate, use at least 59 samples.
Open the parameter-distribution editor and enable the parameters to be sampled. Review distribution types, bounds, and help text before running. Use Edit parameter catalogue when the JSON catalogue itself needs to be changed.
Use Fast mode for large sample sets unless individual plots are specifically needed.
Click Run UA and monitor the status area and run folder.
7.3 Results and Interpretation
After a UA run completes, the Results tab organises the output into three sub-tabs: FOM Distribution, Importance Analysis, and Ordered Statistic. A unit-system selector at the top of the Results tab applies metric or English conversions to all displayed values. A variable selector chooses the FLARE scalar figure of merit (e.g. peak hot-pin clad temperature) and, when FLARECON results are present, a separate containment FOM selector is shown. The sub-tabs described below apply to both FLARE and FLARECON outputs.
FOM Distribution
The FOM Distribution sub-tab shows the time-series overlay for the selected variable (all N samples plotted together with colour coding from lowest to highest), a histogram of the scalar FOM across all samples, a top-10 ordered values table, a Wilks confidence summary table (showing the confidence level beta for order statistics m = 1, 2, 3, 5, and 10 at coverage gamma = 95%), and an empirical CDF. When FLARECON outputs are present, a containment time-series overlay with its own variable selector, a containment scalar summary showing the maximum, mean, and minimum across samples, and H2 colour coding relative to the lower flammability limit (4 vol%) and detonable threshold (8 vol%) are also displayed. A caption directs the user to the Importance Analysis and Ordered Statistic tabs for the complete statistical synthesis.
Importance Analysis
The Importance Analysis sub-tab applies the stepwise multiple-regression method described by Martin (Nucl. Technol. 175, 2011) to decompose the total output variance into contributions from individual sampled inputs. The theory is documented in Section 8.10 of the Code Theory Manual. The user selects the Student-t confidence level (defaulting to the two-sided 97.5% level, which is equivalent to the one-sided 95% level used for Wilks). FLARE computes the minimum correlation threshold r_min from that level and the sample size n, using Eq. (8.35) of the Code Theory Manual.
Four key performance indicator tiles are displayed: the actual standard deviation of the selected FOM across all samples (sigma_o), the convolution estimate of the standard deviation from the accepted contributors (sqrt(Sum Var), Eq. 11 of Martin 2011), the ratio of the convolution estimate to the actual standard deviation (Eq. 12), and the residual standard deviation not explained by the accepted contributors. A ratio of 0.90 or above is generally satisfactory. A ratio below 0.90 indicates that additional contributors may exist below the stopping threshold, suggesting a larger sample size.
The stepwise regression table lists each accepted contributor in order of importance with its Pearson correlation coefficient, delta-standard-deviation contribution, polynomial fit degree (1 or 2, selected by adjusted R-squared), and adjusted R-squared. A terminal stopping row is included to show which parameter caused the regression to halt and its correlation relative to the threshold. A tornado bar chart ranks the accepted contributors by delta-standard-deviation. A second bar chart shows the absolute correlation for each contributor against the threshold line. Scatter plots with the polynomial fit curve are shown for each accepted contributor, with residual output on the vertical axis for steps after the first.
When FLARECON outputs are present, a FLARECON Importance Analysis section appears below the FLARE section within the same sub-tab. It repeats the same analysis for the selected containment FOM using an independent Student-t threshold control. Each FLARECON scalar (containment pressure, containment gas temperature, H2 concentration, sump level) is analysed separately when present.
Ordered Statistic
The Ordered Statistic sub-tab displays the top-20 ordered values as a connected scatter plot with a user-selectable order statistic m highlighted by a star marker. Three summary tiles show the m-th order-statistic value, the Wilks confidence beta at coverage 95% for the current n and m, and the maximum observed value (m = 1). The Wilks confidence tile is coloured green when beta is at or above 95% (indicating that the 95/95 criterion is met) and red when below. A curve of beta versus m (for m = 1 through 10) is shown alongside the ordered values plot, with a dotted reference line at beta = 95%. A minimum-n reference table at the bottom of the sub-tab shows the sample size required to achieve 95/95 for m = 1, 2, 3, 10, 40, and 100.
When FLARECON outputs are present, a FLARECON Ordered Statistics section appears below the FLARE section. It applies the same ordered-statistic and Wilks analysis to each selected containment FOM. For H2 concentration, horizontal dashed reference lines at the lower flammability limit (4 vol%) and the detonable threshold (8 vol%) are overlaid on the ordered values plot.
AI Uncertainty Narrative
An AI Uncertainty Narrative expander appears at the bottom of the Results tab. The narrative is section-by-section (matching the same generation architecture used in the PWR Simulator) and draws on the full statistical results: the selected FLARE and FLARECON FOMs, the sample population summaries, the complete stepwise regression table, the Wilks order-statistic results for m = 1, 2, and 3, Pearson cross-check correlations, and the sampled input matrix. The narrative detail slider controls the number of sections generated and the word target for each, following the same 21-level scale used in the PWR Simulator. At detail levels below 0.15, a compact two-section abstract is generated. At higher detail levels the narrative is expanded to include dedicated sections on the FOM distribution and ordered statistics, importance analysis and uncertainty drivers, FLARECON containment FOM results when present, and a GDC support conclusion. A Clear button discards the stored narrative without affecting the statistical results. The narrative can be downloaded as a Markdown file.
Note: For Wilks use, the most important results are the sorted scalar figures-of-merit and the selected order statistic displayed in the Ordered Statistic sub-tab. Confirm that the sampling is independent and that no samples were rejected due to input combinations outside the valid model range before invoking a Wilks tolerance claim. The importance analysis is a post-processing diagnostic; it does not affect the Wilks statistical interpretation of the output distribution.
7.4 Durable Worker Execution
UA runs execute through flare_ua_worker.py. The UI writes a configuration JSON, starts the worker, and monitors a status JSON file in a timestamped ua_{case}_{timestamp} folder. This prevents browser refreshes from terminating the calculation. If a UA run stops early, inspect the per-sample log associated with the first failed sample. Many failures are caused by an input realization that pushes a property lookup, timestep, or correlation outside its valid range.

Chapter 8 — Risk Assessment
The Risk Assessment tool maps FLARE case results onto a frequency-consequence chart. It is designed for rapid comparison of event categories and mitigation options, not for a complete probabilistic risk assessment. The user supplies event frequencies, FLARE calculates consequences using the NOTBADTRAD dose model, and the tool plots each case relative to the NEI 18-04-style design objective boundary.
8.1 Overview
Each point on the F-C chart represents a complete FLARE case. The horizontal coordinate is consequence, expressed as total EAB TEDE. The vertical coordinate is event frequency, supplied by the user and stored in Runtime\flare_risk_frequencies.json. The total EAB dose plotted is the sum of the accident source-term dose and the iodine-spike dose:
D_total,EAB = D_accident,EAB + D_iodine-spike,EAB
8.2 Setting Up the Case Library
The Risk tool discovers CaseName_in.xlsx files in the selected Working Folder and allows the user to assign a frequency to each case. Frequencies are saved in Runtime\flare_risk_frequencies.json, and a Working Folder copy of flare_risk_frequencies.json is copied into Runtime\ when that folder is selected. Cases used for risk plotting should include the NBT parameters needed for dose calculation. Run each case individually before adding it to a risk batch so that input problems do not obscure risk-tool behavior.
8.3 Running and Reviewing Results
Click Run Risk Model to execute the selected case library. The UI shows progress and per-case console logs. The resulting F-C chart displays points against the design objective boundary. A case below and to the left of the boundary is favorable relative to the screening curve; a case above or to the right should be examined in more detail.
An AI Risk Narrative can be generated from the results using the narrative expander below the F-C chart. The narrative uses the same section-by-section generation approach as the PWR Simulator: each section is produced by a separate API call and displayed as it completes. The detail slider (21 levels, 0.00 to 1.00) controls the target length and the number of sections. At low detail the narrative summarises the event set and the dominant F-C chart contributors in a short report. At high detail it expands to cover each event category, margin relative to the design objective boundary, frequency and consequence uncertainty, and risk-informed conclusions. The narrative can be downloaded as a Markdown file.

Chapter 9 — Case Setup Guide
This chapter provides practical guidance for building useful FLARE cases. It is organized by event class. Each section explains the physical intent of the case, identifies the key input parameters, gives a representative input block, and describes what the user should check after the first run. The example values are starting points, not validated plant-specific inputs.
9.1 General Case-Building Workflow
Choose a descriptive case name such as CaseSBLOCA_2in_HPSI or CaseLOHS_PORV.
Set the initial power, pressure, temperature, inventory, core geometry, and safety-system availability.
Define the initiating event: break size, pump trip, reactivity insertion, loss of heat sink, boron dilution, or other forcing.
Run the case in the PWR Simulator with figures enabled.
Review sequence-of-events timing, pressure response, inventory trend, core cooling, source-term indicators, and any containment response.
Only after the baseline behaves as intended, use the case in UA, Risk, or Run All Cases.
9.2 LOCA Cases
A LOCA case is defined primarily by break size, break timing, system pressure, ECCS availability, accumulator behavior, and decay heat. For a large-break LOCA, the dominant early behavior is depressurization and inventory loss. For a small-break LOCA, the timing of accumulator injection, HPSI, LPSI, PORV behavior, and steam-generator heat removal may be more important.
diameter_break    = 0.22        # m, equivalent break diameter
t_break           = 0.0         # s, break opening time
pressure_kPa      = 14820       # kPa
total_power       = 575         # MWt
core_flag         = 1
hpsi_mdot_rated   = 40.0        # kg/s
lpsi_mdot_rated   = 80.0        # kg/s
Slip_model        = 'Fauske'    # default LOCA slip option
After the first LOCA run, confirm that break flow starts at the intended time, RCS pressure decreases plausibly, accumulator injection begins when RCS pressure falls below accumulator pressure, pumped SI is available after ESFAS and actuation delays, and vessel level and core temperature trends are consistent with the scenario. If PCT rises unexpectedly, inspect break flow, vessel level, core HTC, DNBR, decay heat, and SI flow together rather than focusing on PCT alone.
Note: The default PORV setpoint is effectively disabled (set near the critical point). For LOCA cases this is often intentional; for overpressure or loss-of-heat-sink cases it is not. Set PORV_setpoint_kPa and PORV_area_m2 explicitly whenever PORV operation is part of the scenario.
9.3 LOHS and Pump Trip Cases
Loss-of-heat-sink and pump-trip cases test the interaction among primary flow coastdown, steam-generator heat removal, pressurizer response, PORV operation, and reactor trip logic. Unlike a LOCA, the primary system may remain intact, so pressure control and heat removal assumptions are especially important.
pump_trip_time     = 0.0
PORV_setpoint_kPa  = 16000
PORV_reseat_kPa    = 15500
PORV_area_m2       = 0.008
scram_on_PORV      = 1
pzr_area           = 3.0
pzr_height         = 17.0
pzr_level_init     = 11.0
sg_nat_frac        = 0.05
Review pressure, PORV mass flow, SG heat removal, pump speed, core power, and pressurizer level. The most common setup mistake is leaving the PORV at the disabling default setpoint. Another common issue is mixing old pressurizer pressure-hold parameters with the newer pressurizer level model; remove deprecated pzr_hold_kPa and pzr_pressure_hold inputs from older cases before interpreting results.
9.4 RIA and Rod Ejection Cases
RIA cases require a time-dependent external reactivity insertion entered in column F of the time-series table as rho_ext in pcm. The timestep should be small enough to resolve the insertion and the prompt power response. The Prompt Jump Approximation (pja_flag = 1) should be enabled for rapid sub-prompt-critical insertions. If prompt-critical behavior is being explored, the case should be treated as a screening demonstration.
ria_flag           = 1
pja_flag           = 1
core_flag          = 1
# Column F contains rho_ext(t) in pcm, peak near t = 0.
After running an RIA case, inspect power, reactivity, fuel temperature, cladding temperature, and trip timing. A smooth-looking output can still miss the peak if the timestep or input table is too coarse. For narrow reactivity pulses, refine the time grid and compare peak power and deposited energy.
9.5 ATWS Cases
An ATWS case represents failure to scram following an anticipated transient. In FLARE, this typically means inhibiting automatic scram by setting full_power_flag = 1. Because ATWS outcomes depend strongly on heat removal, pressure relief, moderator feedback, and assumed operator or system action, document all model flags and boundary conditions clearly.
full_power_flag    = 1          # hold power / inhibit automatic scram
sg_flag            = 0          # example: loss of SG heat removal
PORV_setpoint_kPa  = 16000
PORV_area_m2       = 0.008
Note: full_power_flag = 1 inhibits all automatic trip setpoints. Use it only when the ATWS scenario explicitly requires holding power. Do not leave it enabled in cases intended to represent normal trip response.
9.6 Boron Dilution Cases
Boron dilution cases use CVCS makeup flow to reduce boron concentration and introduce positive reactivity. These cases are sensitive to the assumed dilution flow, initial boron concentration, and reactivity coefficient. FLARE treats the effect as a reduced-order reactivity forcing problem.
cvcs_kgs            = 5.0
cvcs_boron_ppm      = 0.0         # unborated makeup
cvcs_start_time_s   = 0.0
alpha_boron_pcm_ppm = -10.0
rcs_boron_ppm_init  = 1200.0      # ppm, initial RCS boron
Check the reactivity history, core power, trip response, and CVCS flow timing. If the response is unexpectedly weak, verify the sign convention for alpha_boron_pcm_ppm and confirm that CVCS flow is active over the intended time interval.
9.7 Dose and Risk Case Inputs
Cases used in the Risk Assessment tool should include the nbt_ parameters described in Section 5.7. Both the consequence assumptions and the frequency basis should be recorded. After the first run, inspect the Source Term, Dose, and Iodine Spike sheets in the output workbook. Confirm that the case produces the intended release category and that zero-release cases are zero because of the modeled event, not because a source-term flag or inventory input is missing.
When benchmarking a case against a published licensing dose (e.g. an FSAR or SSAR Chapter 15 result), set nbt_burnup_gwdmtu to the plant-specific burnup, set nbt_chi_q_eab and nbt_chi_q_lpz to the site PAVAN values for the same receptor distances and time periods used in the licensing calculation, and disable spray credit (nbt_sprays_on = False) unless the licensing calculation explicitly takes spray credit. Differences that remain after matching these inputs are generally attributable to the group-level, NUREG-1465-based source term and FGR-weighted dose conversion factors, and should not be removed with nbt_calibrate without documenting the comparison.
9.8 Adding a FLARECON Containment Model
To add containment response analysis, add a FLARECON sheet to the input workbook. The minimum model defines free volume, floor area, wall area, initial pressure, initial temperature, wall temperature, run time, timestep, and source mode.
DESIGN PARAMETERS

floor_area_m2       = 435.6     # floor area [m²]
wall_surf_area_m2   = 2585      # liner/wall area [m²]
INITIAL CONDITIONS
P0_kPa              = 101.325
T0_C                = 27.0
T_wall0_C           = 27.0
RUN CONTROL AND SOURCE OPTIONS

timestep            = 2.0

SPRAY OPTIONS
spray_from_flare_nbt = True
spray_effectiveness  = 0.8
Review containment pressure, gas temperature, wall temperature, sump level, hydrogen concentration, steam concentration, and the Shapiro diagram. For long containment runs, use the Plant Analyzer Step skip control to accelerate visual playback.
9.9 First-Run Acceptance Checks
Check | Question to answer
Initial state | Do pressure, temperature, power, inventory, and levels match the intended plant condition?
Initiating event | Does the break, pump trip, reactivity insertion, or heat-sink loss occur at the intended time?

10.4 DNBR Output Interpretation
DNBR and CHF Heat Flux are set to NaN or blank when core mass flux is below the validity range of the CHF correlations. This commonly occurs during pump coastdown or natural-circulation conditions. A NaN DNBR in that region does not mean DNB has occurred; it means the correlation is not being applied. Review cladding temperature and heat-transfer regime to interpret the cooling state.
10.5 Conservation and Consistency Checks
Useful consistency checks include total mass scaled, vessel collapsed level, break flow, injection flow, core power, steam-generator heat removal, and containment source mdot. For integrated cases, compare RCS break flow with FLARECON Source mdot to confirm the containment run is reading the expected source file. Remember that FLARECON is one-way coupled; differences due to containment back-pressure feedback are not represented.
10.6 Appropriate Use of Results
When using FLARE results externally, report the case name, input workbook version, major model flags, timestep, source-term option, containment option, and any sidebar overrides. FLARE results are most useful for scoping, sensitivity analysis, uncertainty demonstrations, risk-informed screening, methods comparison, and educational use. Avoid presenting FLARE output as a replacement for a plant-specific, validated evaluation model.
Chapter 11 — Code Assessment and Model Limitations
This chapter summarizes user-level consistency checks and the principal reduced-order limitations. It is not a formal code validation report. Instead, it helps users determine whether a calculation is internally consistent and whether the selected model is appropriate for the question being asked.
11.1 Purpose of User-Level Assessment Checks
A successful run only proves that the software completed and produced output. The user still must determine whether the input represents the intended scenario and whether the output is physically plausible. Basic checks include confirming event timing, latching signals, mass and energy trends, source-term activation, and containment response. These checks should be performed before using a case in batch processing or external reporting.
11.2 Principal Reduced-Order Limitations
The RCS model is a single homogeneous-equilibrium control volume. It cannot resolve axial void distribution, counter-current flow limitation, hot-leg condensation, multidimensional vessel refill, detailed reflood, or local subchannel effects. The hot-channel, rod-state, oxidation, source-term, and dose models are reduced-order post-processors driven by the lumped thermal-hydraulic solution. FLARECON is also single-node and one-way coupled. These simplifications are appropriate for screening and methods demonstrations but not for replacing detailed licensing models.
Model | Limitation
RCS thermal-hydraulics | Single homogeneous-equilibrium node. No axial resolution, no counter-current flow, no hot-leg condensation.
Break flow | Fauske/HEM/Moody critical-flow correlation. No stratification, no pipe flow dynamics.
Core heat transfer | Single slab per pin (average and hot pin). No subchannel, no reflood model.
Decay heat | ANS-5.1-2014 correlation. Column C override replaces it entirely.
Pump model | Reduced affinity-law homologous pump with exponential coastdown. No reverse flow.
FLARECON | Single well-mixed gas node. No local stratification, no flame propagation, one-way coupled from RCS.
Dose model (NOTBADTRAD) | Embedded Pasquill-Gifford dispersion (stability classes A–F) with independent EAB/LPZ/control-room integration periods, or direct site-specific χ/Q override (nbt_chi_q_eab, nbt_chi_q_lpz, nbt_chi_q_cr). No ARCON96/PAVAN-equivalent processing of joint-frequency meteorological data and no time-varying meteorology.
Source term | RG 1.183 group release fractions scaled by reduced-order failure model and by nbt_burnup_gwdmtu. Group dose conversion factors are FGR-11/FGR-12 activity-weighted constants. No nuclide-level transport or time-evolving dose conversion factors.
Table 11-1. Principal model limitations.
11.3 Appropriate Use Boundaries
Use FLARE when transparency, speed, trend comparison, and sensitivity exploration are more important than spatial fidelity. Do not use FLARE where the conclusion depends on local multidimensional flow, detailed fuel mechanics, detailed nuclide transport, site-specific meteorology, or dynamic coupling between containment pressure and RCS break flow. When communicating results, state the reduced-order nature of the model plainly.
Chapter 12 — Troubleshooting
The fastest troubleshooting path is usually to reproduce the problem with a single case, inspect the console log or run-folder log, and then simplify the input until the failing behavior is isolated.

12.2 Tips for LOHS and Pump Trip Cases
Set PORV_setpoint_kPa and PORV_area_m2 explicitly.
Enable a pressurizer model (pzr_area, pzr_height, pzr_level_init) when pressure-control behavior matters.
Use scram_on_PORV = 1 only when PORV opening is intended to initiate scram.
Set sg_nat_frac to represent post-trip natural-circulation heat removal.
Set sg_trip_isolation_enabled = 1 (default) to isolate forced SG flow after reactor trip while retaining natural circulation.
Remove deprecated pzr_hold_kPa and pzr_pressure_hold inputs from older input files.
12.3 Tips for LOCA Cases
Use the intended slip model and document the choice (Fauske for LOCA is the default).
Set HPSI and LPSI rated flows deliberately; defaults of zero disable pumped injection.
Confirm accumulator gas volume, total volume, and initial inventory before interpreting refill timing.
Review vessel level and heat-transfer regime together with PCT.
For cold-leg breaks, consider post_chf_model = 'cold_leg' to represent film boiling at flow stagnation.
12.4 Tips for FLARECON Cases
Use endtime = 86400 s (24 hours) for long-term containment heat-up studies.
Use Step skip in the Plant Analyzer for long containment transients.
Confirm source_mode before interpreting post-blowdown steam addition.
Review both hydrogen concentration and steam concentration before interpreting flammability status.
Use the Shapiro diagram to understand the mixture trajectory, not only the peak value.
Check that air_volume_m3 and wall_surf_area_m2 are consistent with the plant design; the defaults are for a large PWR.
Chapter 13 — FLARECON Containment Model
This chapter gives the user-facing description of the FLARECON containment model. It explains what FLARECON calculates, how it is connected to the RCS output, which inputs are most important, and how to interpret the containment output products. The mathematical theory is documented in the Code Theory Manual.
13.1 Overview
FLARECON is a single-node dry-containment response simulator. It tracks transient pressure, gas temperature, wall temperature, sump temperature, steam concentration, hydrogen concentration, heat removal, and sump accumulation. It runs automatically when the input workbook contains a sheet named FLARECON and produces CaseName-CON.csv.
13.2 What FLARECON Models
FLARECON tracks dry air, steam, and hydrogen masses in a well-mixed containment gas volume. RCS break flow and enthalpy provide the principal source during blowdown. Wall and floor heat transfer, condensation, containment spray, and optional long-term heat removal remove energy. Hydrogen generated by zirconium oxidation in the RCS calculation is injected as a noncondensable gas source.
Wall condensation can use either the Uchida (1965) empirical heat-transfer model or the Diffusion Layer Model with Film/Mist (DLM-FM). The wall model includes a multi-layer finite-difference solver for the steel liner, an optional gap between liner and concrete, and a concrete backing. Spray removal is represented by a Ranz-Marshall droplet heat-transfer model scaled by a lumped effectiveness parameter.
Sump accumulation is tracked by a reduced-order model that distinguishes engineered sump capacity and floor flood depth. Condensate draining from walls wets the full containment floor before standing liquid accumulates to sump overflow.

13.5 Hydrogen Flammability Assessment
FLARECON evaluates the steam-air-hydrogen mixture against a Shapiro-style flammability envelope. The output status is reported as non-flammable or steam-inerted, deflagration region, or detonation region. The model flags envelope entry but does not simulate combustion, flame acceleration, pressure spikes, or thermal loads from hydrogen burning. The Shapiro diagram in the PWR Simulator Containment Response tab shows the trajectory of the mixture over time.
13.6 Limitations
The gas volume is single-node and well-mixed. Local hydrogen stratification and local hot spots are not represented.
Containment pressure does not feed back to RCS break flow.
Spray behavior is represented by a lumped droplet effectiveness model, not spray ring geometry.
Hydrogen combustion is assessed but not dynamically simulated.
LHSI/RHR sump cooling is not explicitly modeled; long-term heat removal is represented by wall conduction and spray only.
Chapter 14 — Worker-Process Execution Architecture
Several FLARE operations are too long or too fragile to run purely as interactive browser tasks. The worker-process architecture lets the UI launch a separate process, monitor status files, and recover useful output even if the browser reloads. This chapter explains what the user should expect from worker-based operations and how to avoid conflicts.
14.1 Why FLARE Uses Workers
Run All Cases, Uncertainty Analysis, and Risk Assessment can take many minutes or longer. A browser tab or Streamlit session can refresh, disconnect, or become stale during that time. Workers reduce that fragility by moving the calculation into an independent process that writes status and output files to a run folder independent of the browser session.
14.2 Common Worker Pattern
Each worker follows the same pattern: create a timestamped output folder, write configuration data, run the required simulations, write per-case logs, update a status JSON file, and check for an abort-request file between major steps. This makes failed runs easier to diagnose because the inputs, logs, and partial outputs are preserved.
Worker | Primary output structure
flare_risk_worker.py | risk_{timestamp}/ with status JSON, per-case logs, output workbooks, and FLARE_risk_results.csv.
flare_ua_worker.py | ua_{case}_{timestamp}/ with status JSON, samples CSV, results CSV, and per-sample logs.
flare_sim_batch_worker.py | sim_{case}_{timestamp}/ folders for each case plus control and status files.
Table 14-1. Worker processes and output structures.
14.3 Monitoring, Abort, and Reload
The UI monitors a worker by polling the status JSON in the run folder. When an abort control is available, selecting it writes an abort-request file that the worker checks between major steps. If the browser appears stale, reload the page and use the previous-run selector where available. Do not assume the worker stopped merely because the UI stopped updating; check the run folder for current output files and the status JSON.
14.4 Concurrency Guidance
Avoid running Risk, UA, and Run All Cases simultaneously from the same FLARE folder. These tools may contend for input files, output names, configuration files, or available memory. For simultaneous studies, copy the FLARE folder to separate working directories and run one major worker process per directory.
