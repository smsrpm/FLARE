# PURPOSE AND SCOPE

FLARE is a compact reactor safety analysis environment for rapid accident-response calculations, sensitivity studies, uncertainty runs, risk-informed screening, and technical demonstrations. It is not a substitute for a plant-specific licensing evaluation model, a formally assessed containment code, or a detailed radiological consequence analysis code. All results should be interpreted in the context of the reduced-order simplifications described in the Code Theory Manual and in Chapter 11 of this manual.

This manual covers installation, tool navigation, input file format, operation of each tool, output interpretation, troubleshooting, and the FLARECON containment model. The companion Code Theory Manual describes the mathematical models.


# CHAPTER 1 — INSTALLATION AND STARTUP

## Requirements

Python 3.11 or 3.12 is required. Use a dedicated virtual environment.

Install all required packages with:
  pip install streamlit plotly openpyxl pandas numpy XSteamPython scipy reportlab anthropic

Package purposes:
- streamlit: web-based UI framework for all FLARE tools
- plotly: interactive time-history plots
- openpyxl: reading and writing Excel input and output workbooks
- pandas: tabular data processing for CSV, UA results, and risk tables
- numpy: numerical arrays and math throughout the simulation engine
- XSteamPython: water and steam thermodynamic properties (IAPWS-IF97)
- scipy: root-finding used in critical-flow and steam-property solvers
- reportlab: PDF report generation for Run All Cases
- anthropic: Anthropic API client for AI Event Narratives and FLARE Assistant (optional but recommended)

The anthropic package enables AI Event Narratives and the FLARE Assistant. Deterministic simulations run without it; narrative features require a valid API key.

## File and Folder Organization

All Python source files and case input workbooks live in a single root FLARE folder. Output files and timestamped run folders are created in the root during execution.

Root FLARE folder (flare-pwr\):
  flare_home.py              -- navigation hub and FLARE Assistant AI chat
  flare_ui.py                -- PWR Simulator user interface
  flare_sim.py               -- primary RCS simulation engine
  flarecon_sim.py            -- FLARECON dry-containment response engine
  flare_model_editor.py      -- Model Editor for RCS and FLARECON inputs
  flare_analyzer.py          -- animated Plant Analyzer
  flare_ua.py                -- Uncertainty Analysis interface
  flare_ua_worker.py         -- UA durable worker process
  flare_risk.py              -- Risk Assessment interface
  flare_risk_worker.py       -- Risk Assessment durable worker
  flare_sim_batch_worker.py  -- durable worker for Run All Cases
  CaseName_in.xlsx           -- user case input files (place here)
  CaseName_out.xlsx / _out.csv  -- RCS output workbook and CSV
  CaseName-CON.csv           -- FLARECON containment output CSV
  sim_CaseName_YYYYMMDD_HHMMSS\  -- timestamped run folder (archived input, output, logs)

Subfolders:
  Examples\    -- supplied example input workbooks; copy to root before running
  Icons\       -- PNG/JPG images used by flare_home.py for tool tiles
  Install\     -- launcher scripts (launch_flare.bat, start_streamlit.ps1, start_ngrok.ps1)
               -- Python virtual environment (flare_env\) created here on first launch
  Manuals\     -- reference copies of Users Manual and Code Theory Manual
  Runtime\     -- flare_config.txt (API key), flare_ua_variables.json (UA catalogue),
               -- flare_ui_params.json (sidebar overrides), .preflight_ok sentinel

The FLARE folder can be moved or renamed without editing any paths. launch_flare.bat locates Install\ relative to itself and Streamlit is launched against flare_home.py in the root.

NOTE: If the FLARE folder is inside OneDrive or another cloud-sync directory, pause sync before running simulations. OneDrive can lock Excel files and cause PermissionError messages.

## Starting FLARE

From the FLARE folder with the virtual environment active:
  streamlit run flare_home.py

Or on Windows, double-click Install\launch_flare.bat. This opens Streamlit at http://localhost:8501 and optionally starts an ngrok tunnel for remote access.

On headless Linux servers, FLARE automatically uses the Agg matplotlib backend. The .preflight_ok sentinel file speeds subsequent launches by skipping the package check.

## Platform Support

Full support: Windows, macOS, Linux desktop. Restricted mode (CLI only, no Streamlit UI, CSV output): iOS (Pyto/Pythonista) and Android (Termux). flare_sim.py can be called directly from the command line in restricted mode.


# CHAPTER 2 — FLARE HOME

The Home page is the navigation and configuration hub. Clicking a tool tile opens that tool in the same browser session. Each tool has a Home button in its sidebar. If a tool file is missing, the Home page shows a user-friendly message.

## FLARE Assistant

The FLARE Assistant is an AI chat interface for answering questions about FLARE models, parameters, output columns, and common workflows. It draws on the Code Theory Manual, User's Manual, and supporting technical material. Use it for questions such as "what does this parameter do," "where is this output produced," or "how should I set up this class of case."

The Assistant requires an Anthropic API key. Preferred approach: enter the key on the FLARE Home sidebar. FLARE writes it to Runtime/flare_config.txt. The file can also be created manually:

  ANTHROPIC_API_KEY = sk-ant-...
  ANTHROPIC_MODEL = claude-sonnet-4-6   # optional; omit to use the default model

NOTE: API use requires an internet connection and may incur charges. Do not commit a personal API key to a shared code repository.


# CHAPTER 3 — MODEL EDITOR

The Model Editor is the recommended way to inspect and modify FLARE input workbooks without directly editing every cell in Excel. It reads CaseName_in.xlsx files, groups parameters into labeled sections, displays tooltips from inline comments, and writes changes back.

## Overview

The editor treats the workbook as a structured model. RCS and FLARECON parameters are displayed separately when both are present. Each group appears as a collapsible section with labeled fields and editable inputs.

## Creating a New Input File

Select New Case in the sidebar, enter a case name (no spaces or special characters). The editor creates CaseName_in.xlsx with a default RCS command block and time-series table. Run the new case once interactively before using it in batch, UA, or Risk runs.

## Editing Existing Parameters

The editor organizes each labeled parameter group as a collapsible section. Tooltips come from inline # comments in the RCS sheet or from description columns in the FLARECON sheet. After editing, click Save RCS Model or Save FLARECON Model.

NOTE: Saving overwrites the original workbook. Create a backup before editing when preserving a baseline is important.

## FLARECON Tab

If the workbook contains a sheet named exactly FLARECON, the Model Editor displays a separate FLARECON Model tab. FLARECON uses plain-text section headers (DESIGN PARAMETERS, INITIAL CONDITIONS, etc.) instead of the #-prefixed headers used in the RCS sheet.

## Import, Resample, and Full Table Editing

Import/Resample reads an existing case and interpolates the time-series table to a new time grid. The Full Table Editor exposes the complete time-series table as an editable dataframe. The time column must start at zero and increase monotonically.


# CHAPTER 4 — PWR SIMULATOR

The PWR Simulator is the primary execution and results-review interface. It runs individual cases, applies temporary sidebar overrides, launches FLARECON automatically when a containment sheet is present, displays calculated time histories and scalar metrics, and generates AI-supported event narratives and batch reports.

## Overview and Typical Workflow

1. Select a case from the Case dropdown.
2. Review input assumptions; apply sidebar overrides only for sensitivity studies.
3. Select Fast mode if figures are not needed.
4. Click Run Simulation. Progress bar shows RCS time advancing; FLARECON status appears after.
5. Review the Results tab. If CaseName-CON.csv was produced, review the Containment Response tab.

At run start, FLARE makes a private temporary copy of the input workbook so the original can remain open in Excel without causing a PermissionError. The archived input is stored in the timestamped run folder.

## Execution Modes, Run Folders, and Workers

Interactive execution runs within the Streamlit session. Run All Cases uses durable workers: the UI writes a configuration file, starts a child process, and monitors status files in a timestamped sim_CaseName_YYYYMMDD_HHMMSS folder. When diagnosing a failed run, inspect the run folder first.

## Results Tab

Scalar metric tiles give a quick screen; time-series plots are essential for physical interpretation. A high PCT should be reviewed together with vessel level, break flow, ECCS injection, DNBR, HTC, and core power to understand the cause. The unit-system selector changes display units but does not alter the CSV data.

## Containment Response Tab

When CaseName-CON.csv is present, a Containment Response tab appears showing peak containment pressure, gas temperature, hydrogen concentration, steam concentration, sump level, and flammability status. The Shapiro diagram shows the mixture path through the flammability envelope.

NOTE: The containment calculation is one-way coupled. RCS output drives FLARECON, but containment back-pressure does not feed back into the RCS break-flow calculation.

## AI Event Narrative and Report Outputs

The AI Event Narrative converts numerical results into a structured, section-by-section technical narrative. Narrative generation is section-by-section: each section is produced by a separate API call and displayed as it completes. A progress bar and status indicator are shown; the page refreshes to the stored final report after all sections complete.

Data passed to the model includes: initial, peak, minimum, and final values for RCS pressure, temperature, core power, vessel level, pump flow, ECCS injection, DNBR, cladding and fuel temperatures, HTC, SG heat removal, equilibrium quality, rod failure counts, source-term group releases, dose values, containment results, and unit-system preference.

The Narrative detail slider controls the number of sections and the target word count for each. The slider has 21 discrete detents from 0.00 to 1.00 in 0.05 increments. Target word count ranges from approximately 300 words at 0.00 to approximately 5,300 words at 1.00.

Section plan by detail level:
- 0.00: plant response overview only (short abstract)
- 0.15: adds safety system and thermal-hydraulic response section
- 0.25: adds fuel integrity and core thermal response section
- 0.40: adds containment response section (when FLARECON data present)
- 0.55: adds source term and radiological consequences section (when data present)
- 0.75: adds review observations and model caveats section
- 1.00: all sections at maximum word target

For benign or verification cases with no fuel damage, FLARE condenses to a short three-section structure regardless of the slider. Sections covering fuel integrity or source term are suppressed when data are absent or zero.

The Build tables only option creates the Sequence of Events and Initial/Final Conditions tables without an API call. Download as Word document creates an individual report. Run All Cases mode compiles individual reports and plots into a final PDF report.

# CHAPTER 5 — INPUT FILE FORMAT

## RCS Command Block

The command block occupies the top rows of the first worksheet. Each row is a Python-style assignment statement. Inline comments begin with # and should document units, modeling intent, and assumptions. The command block ends when the first numeric value appears in column A (start of the time-series table).

FLARE parses the command block via Python exec() in an isolated namespace. Common formatting issues are handled automatically: Unicode minus signs converted to ASCII, trailing unit annotations stripped, thousands-separator commas removed.

Example:
  # CaseLBLOCA - Large Break LOCA from full power
  diameter_break = 0.22          # m, cold-leg break
  pressure_kPa   = 14820         # kPa, initial pressure
  total_power    = 575           # MWt
  core_flag      = 1

## Time-Series Table

The table begins after the command block with a header row whose first cell starts with "Time". Up to three blank or non-numeric rows are tolerated before parsing stops.

Column A: Time (s)         -- simulation time; must start at zero, monotonically increasing
Column B: Structure Q [MW] -- heat-structure input when cond_flag is enabled
Column C: Decay Heat [MW]  -- override for ANS-5.1 decay heat; zero = use built-in
Column D: SG Q [MW]        -- SG heat removal when sg_table_flag=1; negative = overcooling (MSLB)
Column E: CVCS Flow [kg/s] -- net CVCS makeup/letdown adder on top of cvcs_kgs steady-state flow
Column F: rho_ext [pcm]    -- external reactivity insertion for RIA or ATWS cases

## FLARECON Sheet

Add a second worksheet named exactly FLARECON to include containment analysis. FLARECON uses plain-text section headers (not # prefixed). The PWR Simulator detects the sheet automatically and runs FLARECON after the RCS simulation.

Minimum FLARECON model:
  DESIGN PARAMETERS
  air_volume_m3 = 10000
  floor_area_m2 = 435.6

  INITIAL CONDITIONS
  P0_kPa = 101.325
  T0_C   = 27.0
  T_wall0_C = 27.0

  RUN CONTROL AND SOURCE OPTIONS
  endtime     = 86400
  timestep    = 2.0
  source_mode = 'flare_then_decay_steam'

## Parameter Classes

Geometry: change only when representing a different plant or scaled model.
Initial conditions: check first when a transient starts from the wrong state.
Boundary conditions: use to define the initiating event and mitigating systems.
Model flags: document flag choices; they alter model behavior fundamentally.
Numerical controls: use to control integration and reporting, not to tune physics.

NOTE: When using sidebar overrides, record important override values in case notes. The workbook is not changed.

# CHAPTER 6 — PLANT ANALYZER

The Plant Analyzer turns FLARE CSV output into an animated visual replay of the plant response. It reads CaseName_out.csv and automatically looks for a matching CaseName-CON.csv in the same directory. It is a post-processor, not a separate calculation. It can replay previous runs without re-executing the simulation.

Launch from the PWR Simulator Results tab or directly:
  streamlit run flare_analyzer.py

Playback controls: First/Last (jump to start/end), Play/Pause, Timestep slider (manual scrub), Speed (delay between frames), Step skip (advance multiple steps per frame for 24-hour runs), Diagram zoom.

Visual indicators:
- Break stem blue: break mass flow active
- PORV stem blue: PORV relief flow active
- SI injection pipe blue: HPSI or LPSI active
- Vessel fill: collapsed liquid level or inventory trend
- Accumulator fill: accumulator liquid inventory
- Pump impeller: dims as pump coasts down
- Containment dome: gas temperature state; badges show current pressure and temperature

The containment dome color scale is tied to absolute temperature, which allows visual comparison across different cases.


# CHAPTER 7 — UNCERTAINTY ANALYSIS

The Uncertainty Analysis tool performs Monte Carlo propagation of selected input uncertainties. It quantifies how uncertain parameters affect scalar figures-of-merit and time-history envelopes, and supports Wilks-style one-sided tolerance interpretations.

## Overview

A UA run starts from a base case and generates N sampled cases by perturbing selected inputs according to distributions specified in Runtime/flare_ua_variables.json. Each sample is run by the simulation engine. Scalar metrics, time histories, sample values, statistics, and visualizations are collected.

## Setting Up a UA Run

1. Select a base case that has already run successfully as a single deterministic case.
2. Choose N samples. For first-order 95/95 Wilks (m=1), use at least N=59.
3. Open the variable-distribution editor, enable variables, review distributions and bounds.
4. Use Fast mode for large sample sets.
5. Click Run UA and monitor the status area and run folder.

## Pre-Flight Validation

The UI validates all enabled variable distributions before launching. Variables with invalid parameters (e.g. uniform with lower >= upper, or normal with sigma <= 0) are blocked with a red error message per variable. The Run UA button is disabled until all issues are resolved. If a variable passes validation but produces an invalid input during sampling, the worker logs a warning, holds that variable constant at its base value for the affected sample, and continues.

## Results Tab — Three Sub-Tabs

After a run completes the Results tab shows three sub-tabs: FOM Distribution, Importance Analysis, and Ordered Statistic. A unit-system selector (metric or English) and a variable selector (FLARE scalar FOM, e.g. peak hot-pin clad temperature) appear at the top. When FLARECON results are present, a separate containment FOM selector is also shown.

### FOM Distribution

Shows:
- Time-series overlay for all N samples (colour coded from lowest to highest)
- Histogram of the scalar FOM across all samples
- Top-10 ordered values table
- Wilks confidence summary: confidence level beta for m = 1, 2, 3, 5, 10 at gamma = 95%
- Empirical CDF

When FLARECON outputs are present: containment time-series overlay with its own variable selector, containment scalar summary tiles (max, mean, min) with H2 colour coding at 4 vol% LFL (lower flammability limit) and 8 vol% detonable threshold.

A caption directs the user to the Importance Analysis and Ordered Statistic tabs for complete statistical synthesis.

### Importance Analysis

Applies the stepwise multiple-regression method of Martin (Nucl. Technol. 175, 2011) to decompose the total output variance into contributions from individual sampled inputs. Theory is in Section 8.10 of the Code Theory Manual.

User selects the Student-t confidence level (default: two-sided 97.5%, equivalent in breadth to the one-sided 95% level used for Wilks). FLARE computes r_min from that level and n: r_min = t_p / sqrt(n - 2 + t_p^2). At the default level with n=59, r_min = 0.256.

Four KPI tiles:
- sigma_o: actual standard deviation of the FOM across all samples
- sqrt(Sum Var): convolution estimate from accepted contributors (Eq. 11, Martin 2011)
- Ratio: convolution estimate / sigma_o (Eq. 12); green >= 0.90, orange < 0.90
- Residual sigma: unexplained standard deviation

A ratio of 0.90 or above is satisfactory. Below 0.90 suggests additional contributors below the stopping threshold or nonlinear interactions; increasing the sample size may help.

Stepwise regression table: each accepted contributor listed in order of importance with:
  - Pearson correlation coefficient (sign shows direction of influence)
  - delta-standard-deviation contribution
  - polynomial fit degree (1 or 2, selected by adjusted R-squared)
  - adjusted R-squared
  - terminal stopping row: shows which parameter caused the regression to halt and its |r| vs. threshold (this row is a diagnostic, not an accepted contributor)

Tornado bar chart: accepted contributors ranked by delta-standard-deviation.
Correlation bar chart: absolute |r| for each accepted contributor with r_min threshold line.
Scatter plots: per-contributor with polynomial fit curve; residual output on vertical axis for steps after the first.

When FLARECON outputs are present, a FLARECON Importance Analysis section appears below the FLARE section. It repeats the same analysis for the selected containment FOM using an independent Student-t threshold control. Each containment scalar (pressure, gas temperature, H2 concentration, sump level) is analysed separately.

### Ordered Statistic

Displays:
- Top-20 ordered values as a connected scatter plot; user-selectable m highlighted by a star marker
- Three wide summary tiles:
    m-th order-statistic value
    Wilks beta at gamma=95% for current n and m (green >= 95%, red < 95%)
    Maximum observed value (m=1)
- Beta vs. m curve (m=1 through 10) with dotted 95% reference line
- Minimum-n reference table for m = 1, 2, 3, 10, 40, 100

When FLARECON outputs are present, a FLARECON Ordered Statistics section appears below. For H2 concentration, horizontal dashed reference lines are shown at 4 vol% LFL and 8 vol% detonable threshold.

NOTE: For Wilks use, the most important results are in the Ordered Statistic sub-tab. Confirm sampling independence and that no samples were rejected due to input combinations outside the valid model range before invoking a Wilks tolerance claim. The importance analysis is a post-processing diagnostic; it does not affect the Wilks statistical interpretation.

### AI Uncertainty Narrative

An AI Uncertainty Narrative expander appears at the bottom of the Results tab. It uses the same section-by-section generation approach as the PWR Simulator. Data drawn on includes: selected FLARE and FLARECON FOMs, sample population summaries, the complete stepwise regression table, Wilks order-statistic results for m = 1, 2, and 3, Pearson cross-check correlations, and the sampled input matrix.

The same 21-level detail slider controls the number of sections and word target. Below detail 0.15, a compact two-section abstract is generated. At higher levels the narrative expands to include FOM distribution and ordered statistics, importance analysis and uncertainty drivers, FLARECON containment results when present, and a GDC support conclusion.

A Clear button discards the stored narrative without affecting the statistical results. The narrative can be downloaded as a Markdown file.

## Durable Worker Execution

UA runs execute through flare_ua_worker.py in a timestamped ua_{case}_{timestamp} folder. Browser refreshes do not terminate the calculation. If a run stops early, inspect the per-sample log for the first failed sample.

# CHAPTER 8 — RISK ASSESSMENT

The Risk Assessment tool maps FLARE case results onto a frequency-consequence (F-C) chart. The user supplies event frequencies; FLARE calculates consequences using NOTBADTRAD; the tool plots each case against the NEI 18-04-style design objective boundary.

## Overview

Each chart point represents a FLARE case. Horizontal axis: consequence = total EAB TEDE = D_accident,EAB + D_iodine-spike,EAB. Vertical axis: event frequency [events/yr] supplied by the user, stored in flare_pra_table.csv.

# CHAPTER 9 — CASE SETUP GUIDE

## General Case-Building Workflow

1. Choose a descriptive case name (no spaces; e.g. CaseSBLOCA_2in_HPSI).
2. Set initial power, pressure, temperature, inventory, core geometry, and safety-system availability.
3. Define the initiating event: break size, pump trip, reactivity insertion, LOHS, boron dilution.
4. Run in the PWR Simulator with figures enabled.
5. Review sequence-of-events timing, pressure, inventory, cooling, source term, and containment.
6. Only after baseline behaves as intended, use in UA, Risk, or Run All Cases.

## LOCA Cases

Key parameters: diameter_break, t_break, pressure_kPa, total_power, core_flag, hpsi_mdot_rated, lpsi_mdot_rated, Slip_model = 'Fauske'.

After first run, confirm: break flow starts at intended time, RCS pressure decreases plausibly, accumulator injection begins when RCS pressure falls below accumulator pressure, pumped SI is available after ESFAS and actuation delays, vessel level and core temperature trends are consistent.

NOTE: The default PORV setpoint is effectively disabled (~22000 kPa). For LOCA cases this is often intentional; for overpressure or LOHS cases it is not. Set PORV_setpoint_kPa and PORV_area_m2 explicitly whenever PORV operation is part of the scenario.

For cold-leg breaks, consider post_chf_model = 'cold_leg' to represent film boiling at flow stagnation.

## LOHS and Pump Trip Cases

Key parameters: pump_trip_time, PORV_setpoint_kPa, PORV_reseat_kPa, PORV_area_m2, scram_on_PORV, pzr_area, pzr_height, pzr_level_init, sg_nat_frac.

Most common setup mistake: leaving the PORV at the disabling default setpoint. Another common issue: mixing old pressurizer pressure-hold parameters with the newer pressurizer level model. Remove deprecated pzr_hold_kPa and pzr_pressure_hold inputs from older cases.

## RIA and Rod Ejection Cases

Enable ria_flag = 1 and pja_flag = 1. Enter rho_ext(t) [pcm] in column F of the time-series table. The timestep must be small enough to resolve the insertion and prompt power response. Review power, reactivity, fuel temperature, cladding temperature, and trip timing.

## ATWS Cases

Use full_power_flag = 1 to inhibit automatic scram. Document all model flags and boundary conditions. full_power_flag = 1 inhibits all automatic trip setpoints -- do not leave it enabled in cases intended to represent normal trip response.

## Boron Dilution Cases

Key parameters: cvcs_kgs, cvcs_boron_ppm = 0 (unborated), cvcs_start_time_s, alpha_boron_pcm_ppm, rcs_boron_ppm_init. Set si_fill_threshold = 1.0 to disable SI pump flow degradation for intact-RCS conditions.

## Dose and Risk Case Inputs

Include nbt_ parameters (Chapter 5). After the first run, inspect the Source Term, Dose, and Iodine Spike sheets in the output workbook. Confirm that zero-release cases are zero because of the modeled event, not because a source-term flag or inventory input is missing.

## Adding a FLARECON Containment Model

Add a FLARECON sheet to the input workbook. Minimum model defines free volume, floor area, wall area, initial conditions, run time, timestep, and source mode. See Chapter 13 for full parameter reference.

## First-Run Acceptance Checks

Initial state: pressure, temperature, power, inventory, and levels match intended plant condition?
Initiating event: break, pump trip, reactivity insertion, or heat-sink loss occurs at intended time?
Protection signals: trip, ESFAS, SI, PORV, spray signals latch as expected?
Mass and energy trends: pressure, break flow, SG heat removal, inventory physically consistent?
Fuel response: PCT, DNBR, ECR, hydrogen trends consistent with cooling history?
Source term and dose: releases present only when intended?
Containment response: pressure, temperature, steam, hydrogen, spray, sump trends make sense together?
Output files: workbook, CSV, CON CSV, logs, plots, reports created where expected?

## DNBR Output Interpretation

DNBR and CHF Heat Flux are NaN or blank when core mass flux is below the CHF correlation validity range. This commonly occurs during pump coastdown or natural-circulation conditions. A NaN DNBR does not mean DNB has occurred; it means the correlation is not being applied. Review cladding temperature and heat-transfer regime instead.

When using FLARE results externally, report: case name, input workbook version, major model flags, timestep, source-term option, containment option, and any sidebar overrides. FLARE results are most useful for scoping, sensitivity analysis, uncertainty demonstrations, risk-informed screening, methods comparison, and educational use.


# CHAPTER 11 — CODE ASSESSMENT AND MODEL LIMITATIONS

## Principal Reduced-Order Limitations

Model                     Limitation
RCS thermal-hydraulics    Single homogeneous-equilibrium node. No axial resolution, no counter-current flow, no hot-leg condensation.
Break flow                Fauske/HEM/Moody critical-flow correlation. No stratification, no pipe flow dynamics.
Core heat transfer        Single slab per pin (average and hot pin). No subchannel, no reflood model.
Decay heat                ANS-5.1 correlation. Column C override replaces it entirely.
Pump model                Reduced affinity-law homologous pump with exponential coastdown. No reverse flow.
FLARECON                  Single well-mixed gas node. No local stratification, no flame propagation, one-way coupled from RCS.
Dose model (NOTBADTRAD)   Gaussian dispersion with fixed meteorology. No site-specific variability.
Source term               RG 1.183 group release fractions scaled by reduced-order failure model. No nuclide-level transport.

## Appropriate Use Boundaries

Use FLARE when transparency, speed, trend comparison, and sensitivity are more important than spatial fidelity. Do not use FLARE where the conclusion depends on local multidimensional flow, detailed fuel mechanics, detailed nuclide transport, site-specific meteorology, or dynamic coupling between containment pressure and RCS break flow.

## Tips for LOHS and Pump Trip Cases

- Set PORV_setpoint_kPa and PORV_area_m2 explicitly.

# CHAPTER 13 — FLARECON CONTAINMENT MODEL

## Overview

FLARECON is a single-node dry-containment response simulator. It tracks transient pressure, gas temperature, wall temperature, sump temperature, steam and hydrogen concentration, heat removal, and sump accumulation. It runs automatically when the input workbook contains a sheet named FLARECON and produces CaseName-CON.csv. The mathematical models are in Chapter 7 of the Code Theory Manual.

## What FLARECON Models

FLARECON tracks dry air, steam, and hydrogen masses in a well-mixed containment gas volume. RCS break flow and enthalpy provide the principal source during blowdown. Wall and floor heat transfer, condensation, containment spray, and optional long-term heat removal remove energy. Hydrogen from zirconium oxidation is injected as a noncondensable gas source.

Wall condensation models: Uchida (1965) empirical heat-transfer model (default) or Diffusion Layer Model with Film/Mist (DLM-FM). The wall model includes a multi-layer finite-difference solver for the steel liner, an optional gap, and a concrete backing. Spray removal uses a Ranz-Marshall droplet heat-transfer model scaled by a lumped effectiveness parameter.

## Source Modes

source_mode                      Description
flare_then_decay_steam           Default integrated mode. RCS break flow during blowdown, then decay-heat steam. Recommended for LOCA cases.
flare_csv                        Uses only RCS CSV break-flow data for the full containment simulation.
decay_steam                      Simplified decay-heat steam source; no RCS CSV needed.

## Hydrogen Flammability Assessment

FLARECON evaluates the steam-air-hydrogen mixture against a Shapiro-style flammability envelope:
  Non-flammable or steam-inerted: H2 < 4 vol% or steam >= 55 vol%
  Deflagration region: 4 to 75 vol% H2 and steam < 55 vol%
  Detonation screening: 18 to 59 vol% H2 and steam < 35 vol%

FLARECON flags envelope entry but does not simulate combustion, flame acceleration, pressure spikes, or thermal loads from hydrogen burning.

## Limitations

- Single well-mixed gas node; no local hydrogen stratification.
- Containment pressure does not feed back to RCS break flow (one-way coupling).
- Spray behavior is lumped effectiveness, not spray ring geometry.
- Hydrogen combustion is assessed but not dynamically simulated.
- Long-term heat removal uses wall conduction and spray only.


# CHAPTER 14 — WORKER-PROCESS EXECUTION ARCHITECTURE

## Why FLARE Uses Workers

Run All Cases, UA, and Risk Assessment can take many minutes. Workers move the calculation into an independent process that writes status and output files independently of the browser session.

## Common Worker Pattern

Each worker: creates a timestamped output folder, writes configuration data, runs simulations, writes per-case logs, updates a status JSON file, and checks for an abort-request file between major steps.

Worker                  Primary output structure
flare_risk_worker       risk_{timestamp}/ -- status JSON, per-case logs, output workbooks, FLARE_risk_results.csv
flare_ua_worker         ua_{case}_{timestamp}/ -- status JSON, samples CSV, results CSV, per-sample logs
flare_sim_batch_worker  sim_{case}_{timestamp}/ folders per case plus control and status files

## Monitoring, Abort, and Reload

The UI monitors the status JSON. When an abort control is available, selecting it writes an abort-request file. If the browser appears stale, reload the page and use the previous-run selector. Check the run folder for current output files and the status JSON before concluding that the worker stopped.

## Concurrency Guidance

Avoid running Risk, UA, and Run All Cases simultaneously from the same FLARE folder. For simultaneous studies, copy the FLARE folder to separate working directories and run one major worker per directory.
